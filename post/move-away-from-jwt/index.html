Move Away From JWT

August 22, 2018


JWT went really hot since introduced and many companies has adopt it and so do we. When I first looked at JWT, the selling points feels so right that can magically solve our problems, but the reality doesn’t seem so.

It took me few weeks to make the transition from old fashion session cookie to JWT authentication, but the process is pretty smooth. We were pretty happy with the result until we want to invalidate a single token. There are many reasons why we want to do that, we want to invalid a compromised token to mitigate potential damage, we want to invalid old token when user state has changed, like password, access permission or log out. With stateless token, there is no easy way to do it. Either you can rotate secret, which will logout all users immediately, or set a short expiration time and wait it to expire. Either way, you don’t have full control of single user session.

The approach we take is make it semi stateful. We save a user id in the payload, and hash some user states into the secret. Next time server receives a token, it looks up database with user id, use user states to generate secret and then validate the token with this secret. So every time user state change will create a new secret and the old token will be invalidated. Well, it does do the job, but wait, why now it looks much like a session cookie?

Well, now the validation process is much like session cookie, why not compare them.

Validation

// session cookie
1. getSessionID
2. findSession

// JWT
1. getToken
2. generateSecretFromStates
3. validate

The first two steps are similar to both but JWT has one more step validate, where we need to run CPU intensive hashing function and the performance is based on how secure hash algorithm we choose.

Session expire

Both of them has an expiration clam, but when a cookie is expired, browser will delete it and won’t be submitted to server, so we don’t need to do any extra validation at all!

Security

It does look more secure with JWT since it is cryptographically sighed, but that is not unique to JWT, because you can sign your session cookie too! However, using JWT instead of cookie is giving up all security features supported by browser like HttpOnly, Secure, SameSite. Both XSS attack and MITM attack can easily steal user token 😈, I want to have some good night sleep not worrying about those problems.

Summary

I’m not saying JWT is bad, but it is bad to use for sessions. Session cookie has it’s own issues, but it is being used and tested for years. I’d like to see new options that can replace it, but until then, it is still the best option to use.
